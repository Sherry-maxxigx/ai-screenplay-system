## 1. 文档目的

本文档用于把当前项目完整移交给新的维护者。文档覆盖四类信息：

- 从零开始搭建运行环境需要安装什么软件、如何配置、如何启动

这份文档面向第一次接手本项目的人来写，因此会尽量使用“先能跑起来，再理解内部结构”的顺序。

## 2. 项目概述

本项目是一个“AI 辅助剧本创作系统”。它采用前后端分离结构，包含自动化剧本工坊、可视化高级编辑器、运行时 AI 参数配置中心，以及若干预留的分析与指纹能力。

从用户视角看，这个系统目前主要提供三条工作链路：

- 自动化工坊：按“核心设定 -> 人物设定 -> 剧情大纲 -> 剧本正文”的顺序分步生成
- 高级编辑器：在现有剧本文本基础上继续编辑，并尝试生成“下一节拍”，同时同步右侧隐性规划图
- 设置中心：调整当前运行时的模型基座、模型名称、温度、Top-p、安全提供商和密钥状态

从技术视角看，这个项目的目标架构是一个多存储、多轨创作的系统：

- MySQL：结构化数据，例如项目、人物、规则
- MongoDB：非结构化文本、版本历史、协同痕迹
- Neo4j：人物、节拍、伏笔等关系图谱
- Milvus：叙事指纹和向量检索
- FastAPI：后端 API
- Vue 3 + Vite + Element Plus + Monaco Editor：前端应用
- Docker / Docker Compose：容器化运行

需要说明的是：当前版本中，已经真正稳定可用的是“自动化工坊”“高级编辑器”“设置中心”这三块主界面，以及基础 API 链路。MySQL / MongoDB / Milvus / Neo4j 的整体设计已经铺开，但部分能力仍属于原型、预留接口或演示实现。

## 3. 交接时最重要的运行结论

### 3.1 当前最容易成功的运行方式

如果只是要把“现在你看到的网页”跑起来给别人看，最简单的方式不是从头构建全部服务，而是直接启动当前已经准备好的两个 Docker 容器：

- 前端容器：`backup-fixed-test-frontend-demo`
- 后端容器：`backup-fixed-test-backend-demo`

启动命令：

```powershell
docker start backup-fixed-test-backend-demo backup-fixed-test-frontend-demo
```

启动后访问：

```text
前端：http://localhost:18080
后端健康检查：http://localhost:18000/health
```

这是目前最适合交接、演示、验收、培训新接手同事的方式。

### 3.2 为什么当前推荐用 18080 / 18000

项目原始 `docker-compose.yml` 中使用的是：

- 前端：`8080`
- 后端：`8000`
- MySQL：`3306`
- MongoDB：`27017`
- Neo4j：`7474 / 7687`
- Milvus：`19530 / 9091`

但在当前机器上，原始端口可能与其他服务冲突，所以演示版本采用了单独容器和不冲突端口：

- 前端映射到 `18080`
- 后端映射到 `18000`

因此，新维护者只要记住一句话：  
“如果只是要看到现在这套网页，优先启动 `backup-fixed-test-backend-demo` 和 `backup-fixed-test-frontend-demo` 两个容器，然后访问 `http://localhost:18080`。”

### 3.3 下次开机后是否自动可用

当前这两个演示容器没有统一设置自动重启策略，因此电脑重启后、Docker Desktop 启动后，通常还需要手动执行一次：

```powershell
docker start backup-fixed-test-backend-demo backup-fixed-test-frontend-demo
```

如果后续要长期交付给别人使用，建议在稳定阶段把容器改成 `restart: unless-stopped` 或统一回到 `docker compose up -d` 的完整部署方式。

## 7. 环境配置与密钥说明

### 7.1 后端配置文件

主要配置位于：

```text

```

这里定义了：

- MySQL 地址
- MongoDB 地址

- Milvus 地址

### 7.2 `.env` 的约定

`config.py` 中使用了 `pydantic-settings`，支持从 `.env` 读取配置，因此更推荐的做法是：

- 不把真实密钥直接写进代码
- 在 `backend` 目录或者项目根目录准备 `.env`
- 用环境变量覆盖默认值

建议新维护者做的第一件安全整改是：

- 把写在代码里的 API Key 移出代码库
- 统一改成 `.env`
- 不再把真实密钥提交到仓库

### 7.3 当前运行时设置文件

当前已经接通并生效的运行时 AI 设置保存在：

```text
backend/runtime_ai_settings.json
```

这个文件记录：

- 当前默认模型提供商
- 不同提供商对应模型
- temperature
- top_p
- 安全提供商
- 输入过的密钥

前端“设置中心”保存后，后端会写入这个文件。

## 8. 网站能实现什么

这一节从产品角度说明当前网页有哪些页面和能力。

### 8.1 自动化工坊

页面入口：

```text
http://localhost:18080/
```

这是当前最主要的剧本生成入口。

#### 页面作用

把“故事核心设定”拆成结构化输入，再逐步生成：

- 人物设定
- 剧情大纲
- 剧本正文

#### 当前页面特点

- 不再只有一个大输入框
- 用户会看到多个引导式输入框
- 必填和选填被区分开
- 页面会自动整理成一段完整需求文本，再发送给后端

#### 当前必填项

- 故事题材
- 背景设定
- 主角设定
- 风格要求

#### 当前可选项

- 核心冲突
- 结局方向
- 补充说明

#### 当前逻辑

- 如果必填项没填完，不能进入下一步
- 如果可选项没填，系统会自动让 AI 进行补全

#### 自动化工坊能看到的效果

- 分步卡片式工作流
- 核心设定输入区
- AI 自动整理后的需求预览
- 一键生成人物设定
- 一键生成剧情大纲
- 一键生成剧本正文
- 全流程保持中文化界面
- 页面视觉已经统一到更高级的蓝青色工作台风格

### 8.2 高级编辑器

页面入口：

```text
http://localhost:18080/editor
```

这是面向“在已有文本上继续创作”的页面。

#### 页面作用

- 编辑剧本文本
- 在现有内容基础上继续生成下一节拍
- 同步右侧隐性规划轨
- 提取叙事指纹入口

#### 当前页面结构

- 顶部标题区和状态按钮
- 中间统计区：当前文本长度、段落数量、图谱状态
- 左侧显性创作轨：Monaco 编辑器
- 右侧隐性规划轨：关系图谱展示区域

#### 生成下一节拍

该按钮会调用后端 `/api/narrative/generate_beat`。

当前版本已经修复了“每次都生成一样的固定文本”的问题。  
现在它会根据当前文本尝试向后续写一场，并把结果追加到当前内容之后。

虽然它仍然属于规则驱动的演示级续写，不是完整的大模型长上下文编剧代理，但已经具备以下效果：

- 不再永远重复同一段
- 会根据当前文本推断主角、最近的道具、下一场场景类型
- 生成后可以同步图谱

#### 隐性规划轨

右侧图谱区域用于展示从文本中提取出的：

- 角色节点
- 剧本节拍节点
- 伏笔节点
- 它们之间的关系连线

当前版本已经修复过一次数据结构问题，避免“只返回连线、没有节点导致整块空白”。

需要说明：

- 这块目前仍然偏演示能力，不是真正生产级的 Neo4j 智能图推理
- 但它已经可以根据文本内容生成稳定的节点和关系，并在编辑或生成后同步刷新

#### 提取叙事指纹

这是进入指纹相关能力的入口。它与后端的 `fingerprint` 接口组联动，用于：

- 原创性检查
- 专业度评估
- 轨道匹配分析
- 版权证明模拟

### 8.3 设置中心

页面入口：

```text
http://localhost:18080/settings
```

这是当前已经真正接通后端的配置页。

#### 设置中心能做什么

- 切换模型基座
- 切换对应模型
- 调整 temperature
- 调整 top_p
- 选择安全提供商
- 输入不同服务的 API Key
- 保存并立即生效
- 查看当前生效快照

#### 当前可选模型提供商

- 智谱 GLM
- OpenAI
- DeepSeek

#### 当前意义

以前这页只是界面，并没有接入业务链路。  
现在已经接通到后端运行时配置文件，保存后会影响主生成链路。

### 8.4 其他页面

代码中还包含但当前不是主交付重点的页面或路由：

## 9. 网站的推荐使用步骤

这一节是给产品、测试、接手开发者都能看懂的“操作说明”。

### 9.1 标准用户操作流程

#### 第一步：进入自动化工坊

打开：

```text
http://localhost:18080/
```

在第一页填写四个必填项：

- 故事题材
- 背景设定
- 主角设定
- 风格要求

例如：

- 故事题材：悬疑科幻电影
- 背景设定：近未来，废弃多年的海上研究站在台风夜突然恢复运行
- 主角设定：女主角林澈，前研究站工程师，事故唯一幸存者
- 风格要求：压迫感强、悬念推进快、结尾带余味

#### 第二步：点击“生成人物设定”

系统会把结构化输入整合后发送给后端，输出人物设定文本。

#### 第三步：点击“生成剧情大纲”

后端会基于核心设定与人物设定生成剧情大纲。

#### 第四步：点击“生成剧本正文”

后端会基于核心设定、人物设定和大纲生成剧本文本。

#### 第五步：进入高级编辑器继续改

打开：

```text
http://localhost:18080/editor
```

把剧本文本放到左侧编辑器中，继续创作。

#### 第六步：点击“生成下一节拍”

系统会在当前文本基础上，续写下一场或下一段推进内容。

#### 第七步：观察右侧隐性规划轨

右侧会根据文本同步人物、节拍、伏笔节点和关系。

#### 第八步：如需调参，进入设置中心

打开：

```text
http://localhost:18080/settings
```

调整模型、温度、Top-p 并保存。

### 9.2 测试时建议的最小输入

在自动化工坊里，填写以下四项即可：

- 故事题材：悬疑科幻电影
- 背景设定：近未来，一座废弃七年的海上气象研究站在台风夜突然恢复运行，并向外界发送求救信号
- 主角设定：女主角林澈，32岁，前研究站系统工程师，也是当年事故唯一幸存者
- 风格要求：压迫感强，悬念推进快，整体偏电影化中文表达，结尾带余味

这样可以较高概率走通当前流程。

## 10. 当前实现效果总结

为了让新维护者快速理解系统“现在真正做到了什么”，这里把效果拆开说明。

### 10.1 已实现且可直接演示的效果

- 首页能打开自动化工坊
- 必填 / 选填核心设定输入已完成交互设计

- 右侧图谱区域可同步
- 设置中心可保存并回读配置
- 设置页会影响当前生成链路

### 10.2 仍然属于演示 / 原型级的部分

## 17. 数据流与调用关系

### 17.1 自动化工坊数据流

用户在前端填写核心设定后，数据流如下：

1. `PipelineView.vue` 收集结构化输入
2. 前端拼装成完整需求文本
3. 调用 `frontend/src/api/ai.js`
4. 请求后端 `backend/app/api/ai.py`
5. 后端读取当前运行时设置
6. 调用对应模型或使用稳定兜底逻辑
7. 返回人物设定 / 大纲 / 剧本结果
8. 前端展示结果并保存到全局状态

### 17.2 高级编辑器数据流

1. 左侧编辑器文本变化
2. 前端调用 `/api/narrative/sync_graph`
3. 后端 `narrative.py` 调用 `neo4j_db.py`
4. 返回节点和连线
5. 前端渲染右侧隐性规划轨

点击“生成下一节拍”时：

1. 前端提交当前全文
2. 后端生成下一段推进文本
3. 前端把新文本追加到编辑器
4. 再次同步图谱

### 17.3 设置中心数据流

1. `SettingsView.vue` 加载时请求 `/api/ai/runtime-settings`
2. 后端从 `runtime_ai_settings.json` 读取配置
3. 用户修改后点击保存
4. 前端 POST 新配置
5. 后端标准化并落盘
6. 后续生成接口读取这一份最新设置

## 18. 新维护者最应该知道的几件事

### 18.1 先学会区分“当前稳定主链路”和“原型能力”

稳定主链路：

- 自动化工坊
- 高级编辑器
- 设置中心

原型或预留能力：

- 深层图推理
- 完整叙事规则引擎
- 更真实的原创性 / 专业度算法
- 多人协同创作

### 18.2 先不要急着清理所有脚本

项目里临时脚本很多，看起来会比较乱。  
交接初期建议先做到：

- 能启动
- 知道主入口
- 知道当前哪些页面可用
- 知道前后端主要调用关系

等熟悉之后，再逐步做脚本归档和目录清理。

### 18.3 第一轮维护建议

建议新维护者接手后优先做这些工作：

- 把 API Key 从代码移到 `.env`
- 整理根目录和 `backend` / `frontend` 的临时脚本
- 为当前主页面补充更完整的接口注释
- 给运行命令写成正式 `README`
- 决定是否继续保留多数据库架构
- 为容器增加自动重启策略

## 19. 常见问题排查

### 19.1 网页打不开

检查顺序：

- Docker Desktop 是否启动
- `docker ps -a` 是否能看到演示容器
- 端口 `18080` 是否被占用
- 前端容器日志是否有异常

### 19.2 后端健康检查失败

访问：

```text
http://localhost:18000/health
```

如果失败，检查：

- 后端容器是否运行
- 容器日志
- 后端代码是否又被改坏

### 19.3 页面能打开，但生成失败

检查：

- 设置中心中当前模型提供商是否已配置密钥
- `runtime_ai_settings.json` 是否被写坏
- 后端是否因为上游模型异常而回退到兜底逻辑

### 19.4 右侧图谱空白

检查：

- 是否有文本输入
- 浏览器控制台是否报错
- `/api/narrative/sync_graph` 是否返回节点和连线

### 19.5 生成内容异常、格式奇怪

检查：

- 当前模型返回是否不稳定
- 清洗函数是否被改动
- 是否走到了演示兜底分支
